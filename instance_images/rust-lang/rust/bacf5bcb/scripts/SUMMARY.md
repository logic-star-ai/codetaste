# Summary

This testing setup provides a fast and lightweight validation environment for the Rust compiler repository. Instead of building the full compiler (which can take hours), it focuses on running the tidy self-test suite which validates the build system's internal testing infrastructure.

## System Dependencies

The following system dependencies are required and are automatically installed:

- **ninja-build**: Build system used by LLVM (installed by setup_shell.sh if missing)
- **Python 3**: Required for the x.py build system (pre-installed)
- **GCC/G++**: C/C++ compiler toolchain (pre-installed)
- **cmake**: Build system configuration tool (pre-installed)
- **pkg-config**: Library configuration tool (pre-installed)
- **libssl-dev**: OpenSSL development libraries (pre-installed)

All dependencies except ninja-build are already available in the Ubuntu 24.04 environment. The setup script only installs ninja-build if it's missing.

## PROJECT Environment

### Language and Build System
- **Language**: Rust (compiler development)
- **Build Tool**: x.py (Python-based bootstrap build system)
- **Configuration**: config.toml (automatically generated)

### Configuration Details

The setup creates a minimal `config.toml` with the following optimizations:

```toml
[llvm]
download-ci-llvm = false      # Don't download pre-built LLVM (may not be available for old commits)
assertions = false            # Disable LLVM assertions for faster execution

[build]
extended = false              # Don't build extended tools (cargo, clippy, etc.)
docs = false                  # Skip documentation generation

[rust]
incremental = true            # Enable incremental compilation
backtrace = false            # Disable backtrace in the compiler
```

This configuration ensures:
1. **Portability**: Works on both HEAD and HEAD~1 (old commits may not have pre-built LLVM available)
2. **Speed**: Minimal build requirements
3. **Reliability**: No dependencies on external CI artifacts

### Directory Structure

- `/testbed/`: Root of the Rust compiler repository
- `/testbed/build/`: Build artifacts (cleaned by git clean -xdff)
- `/testbed/config.toml`: Build configuration (regenerated by setup_shell.sh)
- `/testbed/src/tools/tidy/`: Source code for the tidy checking tool

## Testing Framework

### Test Suite: Tidy Self-Tests

The test suite runs `tidyselftest` which validates the tidy tool itself. Tidy is Rust's internal linting and consistency checking tool that verifies:

- Code style compliance
- File organization
- License headers
- Dependency management
- Test structure

### Test Execution

**Command**: `python3 x.py test tidyselftest --stage 0`

**Components tested**:
1. **Unit tests** (5 tests): Core tidy library functionality
2. **Binary tests** (0 tests): Main tidy executable tests
3. **Doc tests** (1 test): Documentation examples

**Total tests**: 6 tests (5 unit + 0 binary + 1 doc)

### Test Execution Time

- **First run** (clean build): ~12-15 seconds
  - Bootstrap compilation: ~0.03s
  - Dependency download and compilation: ~12s
  - Test execution: <1s

- **Subsequent runs** (with cache): ~1-2 seconds

### Test Output Format

The run_tests script outputs JSON in the format:
```json
{"passed": 6, "failed": 0, "skipped": 0, "total": 6}
```

This format provides:
- `passed`: Number of tests that passed
- `failed`: Number of tests that failed
- `skipped`: Number of tests that were ignored/skipped
- `total`: Total number of tests executed

## Additional Notes

### Why Tidy Tests?

The tidy self-tests were chosen because they:

1. **Fast**: Complete in ~12 seconds on first run, ~1 second on subsequent runs
2. **Self-contained**: Don't require building the full Rust compiler or LLVM
3. **Representative**: Test the core build system infrastructure
4. **Stable**: Core functionality that rarely changes
5. **Portable**: Work on any commit without requiring CI artifacts

### Compatibility Considerations

The scripts are designed to work on both:
- Current commit (HEAD)
- Previous commit (HEAD~1)

This is achieved by:
- Not relying on `download-ci-llvm` which requires recent CI builds
- Using minimal configuration options
- Only testing self-contained components
- Generating fresh config on each run

### Limitations

These tests validate the build system infrastructure but do NOT:
- Test the Rust compiler itself
- Run full compiler test suites (ui, mir-opt, codegen, etc.)
- Test standard library functionality
- Validate cross-compilation capabilities

For comprehensive testing, additional test suites would need to be run, which would require:
- Building LLVM (30-60 minutes)
- Building stage 1 compiler (30-60 minutes)
- Running compiler test suites (varies by suite)

### Script Idempotency

All scripts are idempotent and safe to run multiple times:
- `setup_system.sh`: No-op (no system services needed)
- `setup_shell.sh`: Only creates config.toml if missing
- `run_tests`: Can be run repeatedly without side effects

The intended workflow is:
```bash
git clean -xdff                    # Clean workspace
sudo /scripts/setup_system.sh     # Setup system (no-op)
source /scripts/setup_shell.sh    # Setup shell and create config
/scripts/run_tests                # Run tests
```

### Environment Cleanup

The `/testbed/` directory is cleaned with `git clean -xdff` before each test run, which removes:
- `build/` - All build artifacts
- `config.toml` - Configuration file
- `__pycache__/` - Python bytecode caches

This ensures each test run starts from a clean state and results are reproducible.
