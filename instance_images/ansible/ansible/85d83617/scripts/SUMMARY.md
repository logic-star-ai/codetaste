# Summary

This repository contains **Ansible**, a Python-based IT automation system. The testing environment has been configured to run the comprehensive unit test suite using pytest.

## System Dependencies

**No system-level dependencies** are required for running Ansible unit tests. The unit tests run in isolation without requiring external services like databases, Redis, or other system services.

## PROJECT Environment

**Primary Language:** Python
**Python Version:** 3.8.20 (via uv-managed Python installation at `/opt/uv-python/cpython-3.8.20-linux-x86_64-gnu/bin/python3.8`)
**Package Manager:** pip (within virtual environment)
**Virtual Environment:** Created at `/testbed/.venv/` using Python 3.8's built-in venv module

### Key Environment Variables
- `PYTHONPATH`: Extended to include `/testbed/lib` for Ansible library modules
- `PATH`: Extended to include `/testbed/bin` for Ansible command-line tools

### Dependencies Installed
1. **Ansible package** - Installed in development/editable mode from `/testbed`
2. **Test requirements** from `/testbed/test/runner/requirements/units.txt`:
   - pytest (test runner)
   - pytest-mock (mocking support)
   - pytest-xdist (parallel test execution)
   - Various cloud provider SDKs (boto3, pyvmomi, etc.)
   - Network automation libraries (netaddr, paramiko)
   - Other testing utilities (mock, passlib, requests, etc.)
3. **ansible-test requirements** from `/testbed/test/runner/requirements/ansible-test.txt`

### Project Structure
- `/testbed/lib/` - Main Ansible library code
- `/testbed/bin/` - Ansible command-line tools
- `/testbed/test/units/` - Unit test suite (~6,500+ tests)
- `/testbed/test/runner/` - ansible-test framework

## Testing Framework

**Test Framework:** pytest
**Test Runner:** pytest (direct execution)
**Test Location:** `/testbed/test/units/`
**Test Count:** Approximately 6,500+ unit tests

### Test Execution
Tests are executed using pytest directly rather than through ansible-test, as the latter creates temporary isolated environments that complicate the setup. The tests are run with the following options:

- `--junit-xml` - Generates JUnit XML output for parsing test results
- `-q` - Quiet mode for less verbose output
- `--tb=no` - Disables traceback display
- `--maxfail=1000` - Continues running tests even after failures (up to 1000)

### Test Exclusions
Some test files are excluded due to collection errors or Python version incompatibilities:
- `test/units/module_utils/test_vmware.py` - Requires Python 3.9+
- `test/units/modules/cloud/vmware/test_vmware_guest.py` - Requires Python 3.9+
- `test/units/plugins/filter/test_mathstuff.py` - Collection error
- `test/units/utils/test_encrypt.py` - Collection error

### Test Output Format
The `/scripts/run_tests` script outputs a single JSON line at the end:
```json
{"passed": <int>, "failed": <int>, "skipped": <int>, "total": <int>}
```

This output is parsed from the JUnit XML file generated by pytest.

## Additional Notes

### Ansible Version Compatibility
This Ansible version (from the repository at the test commits) supports Python 2.7, 3.5, 3.6, 3.7, and 3.8. Since the environment has Python 3.8 available through uv, we use that as the maximum supported version for the most compatible test execution.

### Test Suite Characteristics
- The full test suite contains thousands of tests covering CLI tools, modules, plugins, and utility functions
- Many tests involve mocking external services and cloud providers
- Some tests have known failures, particularly those related to galaxy role initialization and VMware modules
- Tests run relatively quickly (most complete within 10-15 minutes)

### Portability
All setup scripts (`/scripts/setup_system.sh`, `/scripts/setup_shell.sh`, and `/scripts/run_tests`) are designed to work across commits. They have been tested and verified to work on both HEAD and HEAD~1.

### Environment Setup Notes
- The virtual environment is created fresh on each `git clean -xdff` invocation
- Dependencies are installed with markers to skip reinstallation on subsequent runs (using `.venv/.ansible_installed` and `.venv/.test_requirements_installed` flag files)
- The setup is idempotent and can be safely run multiple times
- All installations happen inside `/testbed/.venv/` which is git-ignored and cleaned by `git clean -xdff`
