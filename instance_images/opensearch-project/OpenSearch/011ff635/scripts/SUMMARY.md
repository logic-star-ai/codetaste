# Summary

This testing environment is configured for OpenSearch, a community-driven, open-source search and analytics suite forked from Elasticsearch. The setup enables running a representative subset of unit tests from the server module to validate core functionality.

## System Dependencies

The following system dependencies are required and installed:

- **Java Development Kit (JDK)**: OpenJDK 11 (specifically version 11.0.29)
  - Installed via: `apt-get install openjdk-11-jdk`
  - Required because OpenSearch requires Java 11 to build and run
  - Set as the default Java version using `update-alternatives`

- **Build Tool**: Gradle 7.6 (provided by the gradle wrapper in the repository)
  - No separate installation needed - uses `./gradlew` wrapper script

## Project Environment

### Language and Build System
- **Primary Language**: Java
- **Build System**: Gradle with Gradle Wrapper (gradlew)
- **Project Type**: Multi-module Gradle project

### Environment Configuration
The `setup_shell.sh` script configures:
- **JAVA_HOME**: Set to `/usr/lib/jvm/java-11-openjdk-amd64`
- **PATH**: Updated to include Java 11 binaries
- **GRADLE_OPTS**: Set to `-Xmx2g -Dorg.gradle.daemon=false` for consistent builds
- **Test Seed**: Fixed to `DEADBEEF` for reproducible test results

### Key Environment Settings
- Java source/target compatibility: Java 11
- Working directory: `/testbed` (the project root)
- Gradle daemon: Disabled for reproducibility
- Parallel execution: Enabled with 4 max workers

## Testing Framework

### Test Framework
OpenSearch uses **JUnit** as its primary testing framework with randomized testing support via the `randomizedtesting-runner` library. Tests are located in the `server/src/test` directory.

### Test Execution Strategy
Given the large number of tests (1548+ test classes in server module alone), the test script runs a **representative subset** of tests covering core functionality:

**Selected Test Areas**:
1. Core version and exception handling (`org.opensearch.*Tests`)
2. Cluster health and administration (`org.opensearch.action.admin.*`)
3. Index operations (`org.opensearch.action.admin.indices.*`)
4. Cluster state management (`org.opensearch.cluster.*`)
5. Metadata handling (`org.opensearch.cluster.metadata.*`)
6. Settings and configuration (`org.opensearch.common.settings.*`)
7. Unit conversions (`org.opensearch.common.unit.*`)
8. Index settings and queries (`org.opensearch.index.*`)
9. Search functionality (`org.opensearch.search.*`)

**Total Tests**: ~367 individual test methods across ~20 test classes
**Execution Time**: Approximately 20-75 seconds (depends on cache state)

### Test Result Format
Test results are parsed from JUnit XML files (TEST-*.xml) generated by Gradle in the `build/test-results/test/` directories. The output format is:
```json
{"passed": <int>, "failed": <int>, "skipped": <int>, "total": <int>}
```

### Test Configuration
- **Test seed**: Fixed to `DEADBEEF` for reproducibility
- **JVM args**: Optimized with `-XX:TieredStopAtLevel=1 -XX:ReservedCodeCacheSize=64m` for faster test execution
- **Parallel execution**: 4 worker threads
- **Gradle daemon**: Disabled to avoid cross-contamination between runs

## Additional Notes

### Portability
The scripts are designed to work on both the current commit (HEAD) and previous commits (HEAD~1), making them suitable for regression testing and bisecting issues.

### Git Working Tree Integrity
All dependency installations and builds are performed in directories that are git-ignored (like `build/`, `.gradle/`). Running `git status` after setup shows a clean working tree with no modifications to tracked files.

### Java Version Warnings
The tests may produce warnings about deprecated `System::setSecurityManager` calls. These are expected and do not affect test results. They occur because OpenSearch's test framework uses SecurityManager for test isolation, a feature being removed in future Java versions.

### Gradle Build Cache
The setup leverages Gradle's build cache for faster subsequent runs. The first run compiles all dependencies and test classes, while subsequent runs can reuse cached artifacts.

### Limitations
- The test subset covers core functionality but is not exhaustive
- Full test suite (all 1548+ test classes) would take significantly longer (hours)
- Some integration tests, packaging tests, and BWC (backward compatibility) tests are not included
- Docker-based tests and external module tests are excluded from this subset
